# Service Desk — UI системы мониторинга заявок

Курсовой проект по дисциплине «Проектирование пользовательского интерфейса».
Тема: **Разработка UI для системы мониторинга заявок Service Desk**.
Основной акцент сделан на UI/UX и frontend; бэкенд минимальный (Django) и
выполняет роль JSON-API.

---

## 1. Стек технологий и обоснование

| Слой | Технология | Обоснование |
|------|------------|-------------|
| Разметка | HTML5 (семантические теги) | `<header>`, `<main>`, `<nav>`, `<section>`, `<article>`, `<footer>`, `<address>`, `<time>`, `<dl>`, `<ol>` — структурируют документ и улучшают доступность. |
| Стили | CSS3 (Custom Properties, Grid, Flexbox) | CSS-переменные — для тёмной/светлой темы; CSS Grid — для адаптивной сетки карточек и форм. |
| Скрипты | JavaScript ES6+ | Без транспилеров и сборщиков, чтобы избежать избыточных зависимостей. |
| UI-фреймворк | Vue.js 3 (через CDN) | Подключается одним `<script>` в шаблоне Django, без сборки. Реактивность через `v-model`, `v-for`, `computed`. |
| Сетевой слой | Fetch API | Стандарт браузера, не требует библиотек (например, axios). |
| Бэкенд | Django 5 | Шаблоны + минимальное JSON-API на `JsonResponse` (без DRF, чтобы оставаться в рамках курсовой). |
| Хранилище | SQLite | Подходит для учебного проекта; миграции Django из коробки. |

Никаких лишних зависимостей: единственный пакет в `requirements.txt` — Django.
Bootstrap, jQuery, axios, webpack — намеренно не используются.

## 4. Соответствие требованиям курсовой

### Глава 2. Проектирование интерфейса

#### 2.1. Целевая аудитория и персонализация

- **Персона**: пользователь корпоративной сети (бухгалтер, менеджер), которому
  нужно быстро оставить обращение в IT-поддержку, и **инженер Service Desk**,
  который мониторит входящие заявки. Дизайн ориентирован на оба сценария.
- **Мои заявки**: кнопка в навигации, выборка по ID, сохранённым в
  `localStorage` под ключом `sd_my_tickets`. ID добавляется автоматически
  после успешного `POST /api/tickets/`.
- **Избранное**: кнопка-звезда на каждой карточке; список ID хранится в
  `localStorage` под ключом `sd_favorites`. Используется для быстрого
  возврата к важным обращениям.
- **Тёмная/светлая тема**: переключатель в шапке; значение сохраняется в
  `localStorage` под ключом `sd_theme`. Inline-скрипт в `<head>` применяет тему
  до загрузки Vue — нет «вспышки» светлой темы при загрузке.

#### 2.2. Сценарии и архитектура

- **User Flow** (карта переходов):

  ```
  [Главная: список заявок]
        │
        ├─► поиск / фильтр (status, priority, category) ─► [Отфильтрованный список]
        │
        ├─► «Новая заявка» ─► [Модальное окно] ─► валидация ─► POST ─► обновление списка
        │                                                              │
        │                                                              └─► ID попадает в «Мои заявки»
        │
        ├─► «Мои заявки» / «Избранное»  (переключение представления без перезагрузки)
        │
        └─► карточка ─► звезда (избранное) / показать телефон / удалить (только свои)
                                                                       │
                                                                       └─► [Модал-подтверждение] ─► DELETE
  ```

- **Модальное окно подачи заявки** открывается поверх главной (без ухода
  со страницы). Реализовано через состояние `createModalOpen` и `<transition>`.

#### 2.3. Прототипирование и дизайн-система

- Сетка построена на CSS Grid:
  - `.container { max-width: 1240px; padding: 0 24px; }`
  - `.ticket-grid { grid-template-columns: repeat(auto-fill, minmax(320px, 1fr)); }`
- Дизайн-токены задаются в `:root` (см. секцию 1 `static/css/styles.css`):
  отступы (`--space-*`), радиусы, тени, шрифты, цвета.
- UI-кит:
  - **Кнопки**: `.btn`, `.btn--primary`, `.btn--danger`, `.btn--ghost`, `.btn--sm`.
  - **Инпуты**: `.search__input`, `.form__control`, `.filter__control` — единый
    радиус, фокус-стиль с обводкой 3px и `outline-offset: 2px`.
  - **Карточки**: `.ticket-card` — заголовок, описание (clamp 3 строки),
    список метаданных, цветовая «полоска слева» по приоритету.
- **Акцентные цвета** обоснованы:
  - Синий `#2563eb` — нейтральный и доверительный, используется для основного
    действия «Новая заявка».
  - Красный `#dc2626` — для деструктивных операций (удаление).
  - Оранжевый `#f97316` — для предупреждений и высокого приоритета.
  - Зелёный `#16a34a` — для статусов «Решена» и метки «Ваша заявка».

#### 2.4. Безопасность интерфейса (UX-аспект)

- **Подтверждение удаления**: открывается отдельный диалог `<role="alertdialog">`
  с явным текстом «#ID Заголовок» — пользователь видит, что удаляет.
- **Маскировка телефона**: номер заявителя выводится как
  `+7 •••••• 67` и раскрывается по явному клику пользователя
  (`Показать` → `<a href="tel:...">`). Это уменьшает риск случайного «слива»
  персональных данных, например, при демонстрации экрана.

### Глава 3. Реализация и тестирование

#### 3.1. Технологический стек
См. п. 1 этого README.

#### 3.2. Клиентская логика и реактивность

- Все обновления DOM идут декларативно через **Vue-директивы**:
  `v-model`, `v-bind` (`:class`, `:style`, `:href`, `:disabled`), `v-for`, `v-if/v-else`, `@click`, `@submit.prevent`.
- В исходниках клиентского приложения **нет ни одного `document.querySelector`,
  `innerHTML`, `appendChild` и т. п.** для бизнес-данных — это можно проверить:

  ```bash
  rg "querySelector|innerHTML|appendChild" static/js/app.js
  # ничего не найдено
  ```

- **Поиск «на лету»** реализован через `computed` свойство `filteredTickets`,
  которое реагирует на изменения `search`, `filters` и `activeView`.
  Vue пересчитывает фильтр только при изменении входных данных —
  не нужно вручную перевешивать обработчики и перерисовывать список.
- Большой список (120+ карточек) рендерится мгновенно, потому что Vue 3
  использует виртуальный DOM и точечно обновляет только изменившиеся узлы.

#### 3.3. Взаимодействие с бэкендом (Django API)

| Метод | URL | Назначение |
|-------|-----|------------|
| GET   | `/api/categories/` | Список категорий |
| GET   | `/api/tickets/`    | Список всех заявок |
| POST  | `/api/tickets/`    | Создать заявку (JSON-тело) |
| GET   | `/api/tickets/<id>/`    | Детали заявки |
| PATCH | `/api/tickets/<id>/`    | Изменить статус/исполнителя |
| DELETE| `/api/tickets/<id>/`    | Удалить заявку |

- Запросы выполняются через `fetch`. Реализован **таймаут 15 секунд**
  через `AbortController`, обёртка `apiRequest()` единообразно обрабатывает
  ошибки.
- **Состояние загрузки**: пока `loading === true`, отрисовывается grid
  скелетон-карточек (`.skeleton-card` с CSS-анимацией shimmer).
- **Состояние ошибки**: при сетевой ошибке появляется блок с текстом и
  кнопкой «Повторить», которая снова дергает `loadTickets()`.

#### 3.4. Безопасность реализации

- **Экранирование данных**: текст из API (`title`, `description`, `author_name`)
  выводится через интерполяцию `[[ ... ]]`. Vue по умолчанию делает
  HTML-escape. Директива `v-html` намеренно **не используется нигде**.
  Это значит, что вставка `<script>alert(1)</script>` в описание попадёт
  в DOM как текст, а не исполнится.
- **Client-side валидация**:
  - HTML5-атрибуты: `required`, `minlength`, `maxlength`, `pattern`, `type="email"`.
  - JavaScript-валидация в `validateForm()` дублирует серверную и пишет
    ошибки в `formErrors`, после чего поля получают класс `.is-invalid`
    и под ними появляется красный текст с описанием проблемы.
- **Server-side валидация** в `_validate_ticket_payload()` дублирует
  клиентскую и возвращает `400` с объектом `{errors: {field: message}}`,
  если данные некорректны. Vue выводит эти ошибки в форме.

#### 3.5. Тестирование и оценка качества

- **Адаптивность**:
  - Mobile-first сетка, breakpoints `640px`, `720px`, `920px`.
  - На смартфонах: текст логотипа скрывается, форма становится в одну колонку,
    меню навигации горизонтально скроллится.
  - Проверка: Chrome DevTools → Device Toolbar (iPhone SE / iPad / Desktop).
- **Производительность рендера** на больших списках:
  - Сидер создаёт 120 заявок. Можно увеличить: `python manage.py seed_demo --count 500`.
  - В DevTools → Performance → Record: при вводе текста в поиск рекалькуляция
    `filteredTickets` укладывается в ~5–15 мс на 500 элементах.
- **Доступность**:
  - Semantic HTML (`<header>`, `<main>`, `<nav>`, `<article>`, `<dl>`, `<time>`, `<address>`).
  - Skip-link на содержимое.
  - `aria-live="polite"` на регионе результатов и `aria-busy` на время загрузки.
  - `:focus-visible` стили для всех интерактивных элементов.
  - `prefers-reduced-motion: reduce` — анимации отключаются.

## 6. Что можно показать на защите

1. Открыть `/` — список заявок загружается с показом скелетон-карточек.
2. Ввести в поиск «принтер» — список фильтруется мгновенно.
3. Включить фильтр по статусу «Новая» — увидеть пересечение фильтров.
4. Нажать «Новая заявка» — открывается модалка, попробовать отправить пустую форму
   (показать клиентскую валидацию), затем заполнить корректно.
5. Найти свою новую заявку, нажать звезду — переключиться на «Избранное».
6. Переключиться на «Мои заявки» — увидеть только что созданную.
7. Сменить статус заявки прямо в карточке через выпадающий список — цвет селектора мгновенно меняется, всплывает toast с подтверждением.
8. Нажать «Удалить» на своей заявке — показать диалог подтверждения.
8. Нажать на «••••• 67» — телефон раскрывается, кнопка превращается в ссылку tel:.
9. Кликнуть иконку темы — переключение тёмной/светлой темы, тема сохранится
   после перезагрузки.
10. Открыть DevTools → DevicesToolbar → iPhone SE — показать адаптивность.
11. Открыть Console — никаких ошибок и предупреждений нет.
